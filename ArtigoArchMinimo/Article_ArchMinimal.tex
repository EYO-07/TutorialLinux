\section{Introdução} 

\hspace{\baselineskip}

O objetivo deste texto é guiar o leitor durante a instalação da distribuição \textbf{Arch Linux}. Caso o leitor não tenha interesse em uma instalação customizada, já existem várias distribuições disponíveis de fácil acesso, como \texttt{Ubuntu}, \texttt{Mint} e \texttt{Fedora}. \\

A distribuição \textbf{Arch-Linux} não é para amadores, lembre-se de tomar as devidas precauções antes de instalar o sistema, como backups de arquivos importantes, mídias de instalação extra do arch e de outros sistemas e até mesmo sistemas operacionais mais estáveis instalados em outras partições. Não é incomum que em uma instalação ou outra, ou mesmo na primeira instalação o sistema fique preso no terminal de recuperação do \texttt{initramfs}. \\

Este tutorial irá guiar o leitor para uma instalação mínima do arch usando o gerenciador de janelas \textbf{i3}. O i3 é um pouco mais difícil de usar que os ambientes gráficos tradicionais, pois requer o aprendizado de atalhos de teclado para usar de maneira eficiente. Isso não é um problema para as gerações que já estão acostumadas com jogos eletrônicos, porém não é indicado para os demais usuários. Se não estiver disposto a usar o sistema operacional com atalhos de teclado \textbf{este tutorial não foi feito para você}.\\

O \textbf{Arch Linux} não é recomendado para usuários Linux de primeira viagem, Arch-Linux ainda é considerado uma distribuição instável. Como a distribuição atualiza de maneira contínua, é provavel que em uma atualização ou outra o usuário tenha que resolver problemas que um usuário inexperiente terá dificuldades para resolver. \\

\begin{lstlisting}
* Qual a diferencial do Arch Linux ?
* Como funciona o instalador do Arch Linux ? 
* Existem distros em Arch Linux com instaladores gráficos ? 
\end{lstlisting}

\vspace{\baselineskip}

O arch linux tem um perfil minimalista, o usuário permanece no instalador apenas o suficiente para criar um sistema linux funcional mínimo, sendo a configuração e a instalação dos outros componentes feito a partir dela.\\

Existem distros baseados em Arch Linux com instaladores gráficos, como \texttt{Endeavouros} ou \texttt{CauchyOS}. Tais versões podem ser interessantes para uma primeira experiência em Arch, apesar de pessoalmente considerar que tais distros não preservam a essência e filosofia do Arch e do próprio Linux. A sensação de construir pode ser vista como uma finalidade em si, se o usuário for adepto da filosofia \textbf{DIY} (Do it yourself, Faça você mesmo) o Arch-Linux é o sistema operacional ideal. \\

%A filosofia do \textbf{Arch Linux} é controle, apesar de ser considerado difícil de instalar, ele não é tão difícil para usuários experiêntes do ambiente linux. A abordagem de instalação do Arch Linux é mais trabalhosa, porém tem suas vantagens. O sistema terá melhor desempenho, por conta do menor uso de recursos de hardware por processos em segundo plano desnecessários e o usuário irá ganhar mais experiência sobre o funcionamento do sistema linux. Embora convincentes, nenhum destes argumentos serão mais efetivos do que a essência da filosofia \textbf{DIY} (\textit{do it yourself}, faça você mesmo), a sensação de construir é uma finalidade em si. \\

Neste tutorial será mostrado o passo-a-passo para a instalação do kernel da distribuição \textbf{Arch}, desde os procedimentos iniciais de instalação até a configuração do servidor gráfico legado \textbf{X11}, o gerenciador de janelas em \texttt{tiles} (blocos) \textbf{i3} e o compositor \textbf{picom}.

\section{Recomendações e Requisitos}

\hspace{\baselineskip}

A primeira recomendação é não usar o arch-linux como único sistema instalado, especialmente se esta for a primeira experiência com o arch-linux. Tenha outros sistemas operacionais instalados e evite instalar o grub pelo arch-linux. O espaço necessário para o arch-linux com i3 é pequeno, não mais que 50 \texttt{GB}. \\

\begin{lstlisting}
* Espaço livre necessário para ...
** funcionamento seguro do sistema operacional windows ?
** funcionamento de sistemas operacionais linux ? 
** sistemas operacionais mínimos como arch-linux ? 
\end{lstlisting}

\vspace{\baselineskip}

É interessante para instalação do arch linux uma fonte de consulta (por exemplo, este tutorial), pois a instalação é feita com base em comandos de terminal. Lembre-se de que a instalação de um sistema operacional é uma operação crítica, pois pode envolver perda de dados e incompatibilidades. Não prossiga para instalação do sistema operacional se houver dúvidas. Faça backups de arquivos importantes. Tenha uma mídia para instalação do sistema operacional original. Se possível, instale o sistema linux em outro disco rígido (SSD/HD) secundário, caso contrário tenha bastante espaço em disco para poder criar novas partições. \\

Se você estiver migrando do windows é necessário verificar o status do \textbf{bitlocker}, do \texttt{Secure Boot} (TPM) e da inicialização rápida. Caso o sistema bitlocker de criptografia esteja ativado no seu dispositivo, obtenha e anote em um papel ou caderno a chave bitlocker do seu dispositivo via sua conta microsoft, isso é muito importante, sem a chave bitlocker não será possível sair do modo de recuperação do bitlocker e você irá perder os dados gravados no dispositivo. \\

Se o bitlocker estiver ativado, opte sempre por suspender temporariamente o bitlocker. Desativar o bitlocker irá descriptografar os dispositivos, essa operação consome tempo e pode levar a perda de dados em caso de falta de energia. Mudanças nas configurações da UEFI/BIOS podem acionar o sistema de segurança, por isso tenha em mãos a chave de recuperação. \\

A conexão via cabo de rede ethernet é a mais confiável, problemas com wifi, especialmente de hardwares antigos como a broadcom, são bem comuns e estão bem documentados. Este tutorial irá oferecer um caminho para instalação, mesmo com hardwares wifi antigos, porém não há garantia que funcione em qualquer hardware e problemas específicos devem ser consultados na wiki oficial (\texttt{https://wiki.archlinux.org/title/Main\_page}).\\

Você irá precisar de uma mídia de instalação, recomendo uma mídia \texttt{usb} com a ferramenta ventoy instalada (\texttt{github.com/ventoy/Ventoy}), pois com ela não é necessário reformatação para utilizar diferentes imagens de instalação (\texttt{.iso}), basta copiar os arquivos (\texttt{.iso}). \\

Também recomendo a pré-formatação usando o \textbf{gparted live} (\texttt{gparted.org}, que também pode ser colocada em uma mídia com ventoy instalado). As etapas de formatação são arriscadas na interface de terminal do arch.\\

As perguntas do texto não são exercícios para serem respondidos formalmente, nem são perguntas que necessariamente serão respondidas aqui. São perguntas para verificação rápida dos conceitos. Não execute operações descritas neste texto enquanto houver conceitos obscuros. Se possível, considere primeiro testar a instalação e o sistema em uma máquina virtual.\\

\vspace{\baselineskip}

Prepare as partições a serem usadas na instalação do arch-linux usando o gparted-live ou usando o gparted em algum outro sistema linux instalado, inclusive pode ser interessante nomear as partições com rótulos apropriados, estes rótulos aparecem quando se usa o comando \texttt{blkid} e irão facilitar a identificação das partições a serem utilizadas.\\

Se a sua partição estiver no formato \texttt{GPT} você irá precisar de uma partição \texttt{ESP} para armazenar os arquivos de inicialização do sistema, você pode utilizar uma partição \texttt{ESP} existente caso já tenha (tome cuidado para não formatá-la acidentalmente), caso não exista você deve criar uma partição com pelo menos 512 \texttt{MB} (recomendável 1 \texttt{GB}) em \texttt{FAT32}.\\

É interessante ter uma partição \texttt{swap} caso queira usar o modo de hibernação. Você pode reaproveitar swaps existentes, porém atente-se ao risco de hibernar um sistema operacional e carregar em outro, caso esteja em uma configuração multi-boot isso pode levar à falhas de inicialização.\\

É suficiente 50-70 \texttt{GB} em \texttt{ext4} para a partição principal do sistema (partição root /). Um sistema arch mínimo com i3 como gerenciador de janelas ocupa pouco espaço, porém outros ambientes desktop podem ser mais pesados, se este for o sistema principal você pode alocar mais espaço como 100 \texttt{GB}, o que pode permitir a instalação de outros ambientes desktops secundários. \\

Baixe a \texttt{.iso} do site oficial do arch-linux (\texttt{https://archlinux.org/download/}) e prepare a mídia de instalação. Se estiver usando o windows, tome as devidas precauções com o bitlocker (tenha a chave de recuperação). \\

Em caso de dúvidas que não forem respondidas neste texto, também recomendo consultar diretamente as instruções oficiais de instalação no site:

$$ \texttt{https://wiki.archlinux.org/title/Installation\_guide} $$

\vspace{\baselineskip}

Diferente de outras distribuições em que o usuário interage com um instalador em um processo linear de instalação, a interface de instalação do arch-linux é um terminal. Para usuários do ambiente linux experientes a instalação é muito mais confortável desta maneira, sendo o processo de instalação feito através de comandos. \\ 

Se o usuário compreender o processo, a instalação via terminal se tornará extremamente flexível. A instalação do seu sistema arch-linux pode ser feita praticamente de maneira não-linear, você pode parar e recomeçar a instalação a qualquer momento. Porém existem etapas que necessariamente precisam ser feitas em ordem.\\

\begin{lstlisting}
1. Preparação das Partições (gparted)
2. Instalação do Sistema Base (pacstrap)
3. Configurações Básicas (arch-chroot)
4. Instalação Personalizada (arch-mínimo)
\end{lstlisting}

\vspace{\baselineskip}

Caso você esteja com conexão via cabo ethernet você provavelmente não terá problemas, porém na instalação por wifi, se o hardware for incompatível com o driver você não conseguirá prosseguir com a instalação, pois o arch linux depende de conexão com a internet. \\

A estratégia para prosseguir com a instalação usando hardware wifi incompatível é bloquear a inicialização dos drivers incompatíveis antes do boot do sistema de instalação. Os drivers que geralmente dão problema são (b43,ssb,bcma,brcmsmac,b43legacy,brcmfmac), porém você deve pesquisar primeiro qual combinação pode resolver o problema para o seu adaptador específico. Vamos supor que baste bloquear o b43, quando estiver no menu de inicialização da imagem de instalação, selecione o boot do arch linux e pressione \texttt{e} ou \texttt{Tab} para editar as variáveis de inicialização.\\

\begin{lstlisting}
# no menu de boot do instalador:
# ... pressione e ou tab para editar as variáveis de inicialização 
# ... coloque no final a variável abaixo
# ... com os nomes dos drivers a serem bloqueados.
modprobe.blacklist=b43,ssb 
\end{lstlisting}

\vspace{\baselineskip}

Para se conectar a uma rede wifi no terminal de instalação do arch-linux, você deve executar \texttt{iwctl}. Esse comando irá abrir um subterminal que você pode executar os seguintes comandos: \\

\begin{lstlisting}
# Inventário : comandos do iwctl { Arch Linux }
# 1. device list ; mostra o nome das interfaces wifi disponíveis 
# 2. station NOME_INTERFACE scan ; ativa a detecção de rede da interface
# 3. station NOME_INTERFACE get-networks ; mostra lista das conexões disponíveis
# 4. station NOME_INTERFACE connect NOME_DA_REDE ; conecta a rede 
# 5. exit ; sai do subterminal iwctl
\end{lstlisting}

\vspace{\baselineskip}

Primeiro entre no ambiente do \texttt{iwctl} usando o comando \texttt{iwctl}. Use o comando \texttt{device list} para mostrar o nome das interfaces disponíveis, vamos supor que a interface wifi seja \texttt{wlan0}, ative a deteção de rede usando \texttt{station wlan0 scan}, mostre a lista de conexões disponíveis usando \texttt{station wlan0 get-networks}. Com o nome das conexões você pode conectar a interface wifi com a rede através do comando \texttt{station wlan0 connect REDE}, irá aparecer um prompt pedindo a chave de segurança da rede, caso seja uma rede wifi fechada. Para sair do ambiente \texttt{iwctl} basta executar o comando \texttt{exit}. \\

Teste a conexão com o comando \texttt{ping archlinux.org} ou \texttt{ping 8.8.8.8}. Ele vai mostrar o intervalo de envio e recebimento de pacotes (pressione \texttt{ctrl+c} para sair). Ter um ping menor que 20 \texttt{milisegundos} é o ideal, pings mais altos podem fazer o instalador falhar. Caso você veja mensagens de erro do instalador, não se preocupe, basta executar o comando de instalação novamente, pois o instalador irá retornar ao ponto que parou dos downloads. \\

Preste bem atenção às mensagens de erro, pois o instalador pode retomar a instalação, mas não irá fazer nenhum milagre, se o download falhar a instalação não irá acontecer e você irá montar um sistema incompleto. Para componentes básicos isso pode obviamente impedir sua inicialização. \\

Agora que você está conectado a internet, você deve checar o layout do teclado, teste os símbolos especiais como as barras inclinadas (slash, backslash), os símbolos de pontuação, os símbolos de parênteses, caso não retorne nada ou retorne de maneira incorreta, significa que você precisará ajustar o layout. Você precisa executar o comando \texttt{loadkeys LAYOUT}, geralmente \texttt{loadkeys br-abnt} funciona para a maior parte dos teclados do brasil, porém se não funcionar use o comando \texttt{localectl list-keymaps} para verificar as opções de layout disponíveis para os teclados brasileiros.\\

\begin{lstlisting}
# Inventário : teclado, rede e horário { Arch Linux }
# 1. localectl list-keymaps ; lista os nomes dos layouts de teclado
# 2. loadkeys br-abnt ; configura layout do teclado para português brasileiro
# 3. ping archlinux.org ; testa a conexão com a internet
# 4. iwctl ; inicia utilitário de configuração de rede sem fio
# 5. timedatectl set-ntp true ; ajusta o relógio automaticamente 
\end{lstlisting}

\subsection{Pacstrap - Primeira Etapa}

\hspace{\baselineskip}

A partir daqui será assumido que as partições já estão pré-formatadas e prontas para instalação. Não se esqueça de ajustar o relógio, isso pode prejudicar o download de pacotes. A instalação mínima do sistema pode ser dividido em três etapas, a primeira etapa é a instalação inicial, usando o kernel e os programas da mídia de instalação, a segunda etapa é a etapa do chroot, que apesar de feita no terminal de instalação, usa o kernel e componentes do sistema base instalado através do comando \texttt{arch-chroot}, e a terceira etapa é a instalação já dentro do sistema operacional do arch-linux recém instalado. \\

\begin{lstlisting}
# Inventário : instalação mínima { Arch Linux }
# -- (1) kernel da mídia de instalação --
# 1. ls /sys/firmware/efi/efivars ; verifica se o boot está em modo UEFI
# 2. blkid ; lista discos e partições disponíveis no sistema
# 3. fdisk -l ; lista discos e partições disponíveis no sistema
# 4. mkswap /dev/sdXY ; inicializa partição de swap
# 5. swapon /dev/sdXY ; ativa a partição de swap
# 6. mount /dev/sdXY /mnt ; monta partição raiz (/) em /mnt
# 7. mount --mkdir /dev/sdXY /mnt/boot ; monta partição de boot
# 8. mount --mkdir /dev/sdXY /mnt/boot/efi ; monta partição ESP
# 9. pacstrap /mnt base linux linux-firmware ... ; instala sistema base
# 10. genfstab -U /mnt >> /mnt/etc/fstab ; gera arquivo de tabela de sistemas de arquivos
\end{lstlisting}

\vspace{\baselineskip}

Tenha muita atenção nesta primeira etapa, se você errar as partições, você pode acabar perdendo dados e apagando sistemas existentes. Execute o comando \texttt{blkid} para verificar os nomes das unidades de disco. Deve aparecer uma lista com os nomes seguidos de informações de identificação, como rótulo, tipo de formatação, números de identificação UUID. Os nomes que aparecem podem mudar, sendo mais confiável utilizar os UUIDs para montagem quando o sistema estiver completo. Porém, no terminal do arch-linux isso é muito pouco prático e propenso a erros. Use o nome das partições consultando sempre que necessário o comando \texttt{blkid}.\\

Verifique o tipo de tabela de partição usando o comando \texttt{ls /sys/firmware/efi/efivars}, se não retornar nada ou retornar erro significa que você está em um disco \texttt{MBR}, caso retorne algo, então você está em um disco com tabela de partição \texttt{GPT}.\\

Suponha que a tabela de partição do disco seja \texttt{MBR} e você tenha duas partições, a partição swap em \texttt{/dev/sda2} com 8 \texttt{GB} e a partição \texttt{/dev/sda3} com 50 \texttt{GB} em \texttt{ext4} que será usado para o root. Para a instalação começar você precisa montar a partição root e ativar o swap. Execute o comando \texttt{mount /dev/sda3 /mnt} para montar a partição root, caso o swap nunca tenha sido utilizado execute \texttt{mkswap /dev/sda2}, caso seja um swap compartilhado já existente você pode executar imediatamente a ativação com \texttt{swapon /dev/sda2}. \\

Verifique se as partições foram montadas corretamente com o comando \texttt{lsblk}. As unidades \texttt{sda2} e \texttt{sda3} devem estar com os pontos de montagem \texttt{swap} e \texttt{/mnt} respectivamente. Se as montagens estiverem incorretas você terá que desmontar as respectivas partições usando os comandos \texttt{umount -R /mnt} para partição root e \texttt{swapoff} para o swap. \\

Se você estiver em um disco com tabela de partição \texttt{GPT} você irá precisar montar a partição \texttt{ESP} além das partições anteriores, supondo que você tenha criado ou esteja utilizando uma partição \texttt{ESP} existente em \texttt{/dev/sda1}, basta usar o comando \texttt{mount --mkdir /dev/sda1 /mnt/boot/efi}. Lembre-se de montar a partição root primeiro que a partição \texttt{ESP}, a inversão desta ordem pode acabar populando os endereços dos arquivos de maneira incorreta. \\

Verifique se as partições foram montadas corretamente com o comando \texttt{lsblk}. Você terá que verificar agora a montagem do \texttt{ESP} da unidade \texttt{/dev/sda1} em \texttt{/mnt/boot/efi}. \\

Com as partições montadas você pode prosseguir para a instalação do sistema base usando o comando \texttt{pacstrap}. O pacstrap em tese pode instalar qualquer pacote de instalação dos repositórios oficiais, porém o recomendado é instalar somente os pacotes essenciais para criar um linux bootável mínimo, isso vai permitir resolver os problemas mais críticos do sistema primeiro. \\

\begin{lstlisting}
# instalação do sistema base e firmwares 
pacstrap /mnt base linux linux-firmware
# firmwares para processadores intel
pacstrap /mnt intel-ucode 
# firmwares para processadores amd
pacstrap /mnt amd-ucode
# programas essenciais 
# 1. sudo : programa de elevação de privilégio sudo 
# 2. nano : editor de texto para terminal 
# 3. networkmanager : ferramenta de configuração de rede 
# 4. grub : utilitários do grub para inicialização do sistema
pacstrap /mnt sudo nano networkmanager grub
\end{lstlisting}

\vspace{\baselineskip}

Use os comandos acima para instalar o sistema base usando \texttt{pacstrap}, você pode instalar ambos os pacotes \texttt{intel-ucode} e \texttt{amd-ucode}, porém é interessante escolher o correspondente ao seu processador. O editor de código e texto \texttt{nano} é mais amigável e será bastante útil para configurar os arquivos iniciais do sistema, porém você pode substituir por outros como o \texttt{vim} ou \texttt{vi}.\\

Verifique se não houve mensagens de erro, você pode executar os comandos novamente em caso de perda de conexão, o pacstrap consegue retornar de maneira inteligente o download dos pacotes. Nessa primeira etapa não será feito a instalação dos firmwares de placas de vídeo como o nvidia, portanto o sinal de vídeo provavelmente estará no monitor da placa-mãe. \\

Para finalizar a primeira etapa de instalação agora execute o comando:

$$ \texttt{genfstab -U /mnt $>>$ /mnt/etc/fstab} $$

\vspace{\baselineskip}

Esse comando gera a montagem automática das unidades montadas na instalação, sem esse comando o sistema não vai montar as partições onde foi instalado o sistema, nem a partição swap e a partição ESP. O argumento \texttt{-U} é muito importante, pois ele vai referenciar as unidades pelo UUID. Erros no fstab impedem a inicialização correta do sistema, por isso é sempre interessante usar ferramentas auxiliares como o \texttt{genfstab}, mas nada impede do usuário editar manualmente o arquivo \texttt{/mnt/etc/fstab}. Use o comando \texttt{cat /mnt/etc/fstab} para verificar o resultado. \\

\begin{lstlisting}
# Inventário : instalação mínima { Arch Linux }
# -- (2) kernel do novo sistema usando chroot --
# 1. arch-chroot /mnt ; alterna raiz para o novo sistema instalado
# 2. ln -sf /usr/share/zoneinfo/Região/Cidade /etc/localtime ; define fuso horário
# 3. hwclock --systohc ; sincroniza relógio de hardware com o sistema
# 4. nano /etc/locale.gen ; edita arquivo para descomentar idioma desejado
# 5. locale-gen ; gera as localizações configuradas
# 6. echo "LANG=pt_BR.UTF-8" > /etc/locale.conf ; define idioma padrão do sistema
# 7. echo "NOME_DO_HOST" > /etc/hostname ; define nome da máquina
# 8. pacman -S networkmanager ; instala gerenciador de rede
# 9. systemctl enable NetworkManager ; habilita serviço de rede na inicialização
# 10. mkinitcpio -P ; regenera imagem de inicialização
# 11. passwd ; define senha para o usuário root
# 12. pacman -S grub efibootmgr ; instala bootloader GRUB e ferramentas UEFI
# 13. grub-install --target=x86_64-efi --efi-directory=/boot --bootloader-id=GRUB ; instala GRUB
# 14. grub-mkconfig -o /boot/grub/grub.cfg ; gera arquivo de configuração do GRUB
# 15. useradd -m -G wheel NOME_USUARIO ; cria novo usuário e adiciona ao grupo wheel
# 16. passwd NOME_USUARIO ; define senha para o novo usuário
# 17. EDITOR=nano visudo ; edita arquivo sudoers para conceder privilégios
# 18. reboot ; reinicia o sistema para o novo ambiente
\end{lstlisting}

\subsection{Arch-Chroot - Segunda Etapa}

\hspace{\baselineskip}

A segunda parte da instalação do sistema arch-linux via terminal é feita usando a ferramenta \texttt{arch-chroot}. Nessa etapa, em vez de usarmos as ferramentas e o kernel da mídia de instalação, iremos usar as ferramentas e os recursos instalados no sistema recém-criado. O comando \texttt{arch-chroot /mnt} configura o terminal como se estivesse dentro do sistema construído, assim os endereços e os comandos ficam relativos ao sistema, não sendo mais necessário referenciar o ponto de montagem do sistema \texttt{/mnt}. Por exemplo, se quisermos saber o conteúdo do \texttt{fstab}, basta digitar \texttt{cat /etc/fstab} em vez de \texttt{cat /mnt/etc/fstab}. \\

Nesta etapa as configurações de teclado, localização, horário, senha do usuário root e usuários iniciais são criados. Sem estas configurações o sistema não é inicializável, portanto o \texttt{arch-chroot} é necessário para fazer estas edições. \\

Para configurar o idioma primeiro você precisa criar e editar o arquivo \texttt{/etc/locale.gen} com os pacotes de idiomas indicados como abaixo: \\

\begin{lstlisting}
# exemplo /etc/locale.gen
pt_BR.UTF-8 UTF-8
en_US.UTF-8 UTF-8
\end{lstlisting}

\vspace{\baselineskip}

Use o editor \textbf{nano}, usando o comando \texttt{nano /etc/locale.gen} e use \texttt{ctrl+o} para salvar e \texttt{ctrl+x} para sair do editor. Criado o arquivo agora basta usar o comando \texttt{locale-gen} para que o sistema instale e configure os idiomas selecionados baseado no que estiver indicado do arquivo.\\

Para selecionar o idioma padrão, você deve configurar a variável \texttt{LANG} no arquivo \texttt{/etc/locale.conf}, crie e edite o arquivo adicionando a linha \texttt{LANG=en\_US.UTF-8} para língua padrão em inglês ou outra linguagem de preferência, como o português \texttt{LANG=pt\_BR.UTF-8}. No arquivo \texttt{locale.conf} também pode ser configurado o padrão do relógio com \texttt{LC\_TIME=...} caso tenha preferência. \\

Para configurar o relógio e horário do sistema você deve usar os comandos abaixo:

\begin{lstlisting}
# 1. Configura o horário pela rede
timedatectl set-ntp true
# 2. Configura o relógio para Brasil região Leste
# ... aperte tab duas vezes com o input 
# ... 'ln -sf /usr/share/zoneinfo/Brazil/'
# ... para aparecer as outras opções.
ln -sf /usr/share/zoneinfo/Brazil/East /etc/localtime 
# 3. Registra o horário no relógio do hardware 
hwclock --systohc 
\end{lstlisting}

\vspace{\baselineskip}

Para configurar o teclado basta editar o arquivo \texttt{/etc/vconsole.conf} que irá editar as configurações do console. Adicione a linha \texttt{KEYMAP=br-abnt} ou \texttt{KEYMAP=...} para a opção que funcione para o layout do seu teclado. \\

Feito isso, execute o comando \texttt{echo "PC\_NAME" > /etc/hostname} com \texttt{PC\_NAME} o nome do computador. Não crie nomes crípticos, não deve ser uma senha, mas um nome humanamente legível para identificar o computador na sua rede. Crie agora o arquivo \texttt{/etc/hosts} conforme abaixo: \\

\begin{lstlisting}
# /etc/hosts
# vamos supor que o hostname seja archmin
127.0.0.1   localhost
::1         localhost
127.0.1.1   archmin.localdomain archmin
\end{lstlisting}

\vspace{\baselineskip}

Execute os seguintes comandos para criar usuários:\\

\begin{lstlisting}
# 1. Mude a senha do usuário root com:
passwd 
# 2. Adicione o usuário NOME ao grupo wheel 
# ... o grupo wheel são usuários que 
# ... podem executar comandos root com sudo
useradd -m -G wheel NOME
passwd NOME 
# 3. Habilite o grupo wheel 
# ... execute o comando abaixo e descomente 
# ... a linha %wheel ALL=(ALL) ALL
EDITOR=nano visudo 
\end{lstlisting}

\vspace{\baselineskip}

O grupo wheel era o grupo tradicional de usuários com privilégios de administrador, quem estiver migrando de sistemas \textbf{Debian} pode estranhar o uso do grupo wheel, pois o \textbf{Debian} adotou o grupo \texttt{sudo}, porém no \textbf{Arch} e nos sistemas tradicionas o grupo \textbf{wheel} tem essa funcionalidade.\\

Para finalizar a segunda etapa de instalação (a última etapa da instalação mínima) é necessário executar comandos para tornar o sistema bootável (inicializável), vamos usar o sistema \texttt{grub}, pois a maior parte das outras distribuições utilizam esse sistema. \\

\begin{lstlisting}
# Inventário : Instalação Grub { Arch-Linux }
# 1. grub-mkconfig -o /boot/grub/grub.cfg ; cria o arquivo de inicialização
# 2. grub-install --target=i386-pc /dev/sdX ; instala e registra o grub na partição MBR 
# 3. pacman -S grub efibootmgr ; instala os pacotes grub e efibootmgr
# 4. grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=GRUB   
# ... instala o grub para partições UEFI/GPT 
\end{lstlisting}

\vspace{\baselineskip}

A execução do comando \texttt{grub-mkconfig -o /boot/grub/grub.cfg} é estritamente necessária, caso contrário o sistema não irá inicializar. A instalação do grub por outro lado não é necessária caso o seu sistema seja secundário. Em uma disposição multi-boot, em que temos vários sistemas operacionais linux, recomendo que o arch-linux seja o sistema secundário, pois o arch-linux é mais instável, assim não reinstale o grub, execute \texttt{update-grub} no terminal do sistema operacional principal para ele detectar o \texttt{arch-linux} automaticamente. \\

Caso você queira o arch-linux como sistema principal, você pode executar os comandos \texttt{grub-install} respectivos para partições GPT ou MBR. Caso você esteja usando a partição GPT, execute o comando para instalar o pacote \texttt{efibootmgr}, do contrário a instalação irá falhar. \\

Em caso de querer tornar o arch-linux o sistema principal (o que não recomendo), você irá precisar instalar o pacote \texttt{os-probe} para detectar outros sistemas operacionais, caso você esteja em tabela de partição GTP você terá que instalar também o pacote texttt{efibootmgr}. Você pode incluir o pacotes no comando do pacstrap (fora do ambiente arch-chroot) ou no comando \texttt{pacman -S os-probe efibootmgr} dentro do ambiente arch-chroot. Porém você deverá habilitar o \texttt{os-probe} editando o arquivo \texttt{/etc/default/grub} e descomentando a linha específica com a variável \texttt{GRUB\_DISABLE\_OS\_PROBER=false} para que a detecção dos sistemas aconteça. Não esqueça de verificar o resultado com o comando \texttt{cat /boot/grub/grub.cfg} para checar se houve detecção dos outros sistemas. \\

\begin{lstlisting}
# Finalização da Segunda Etapa
mkinitcpio -P # atualiza o initramfs 
exit # sai do arch-chroot 
umount -R /mnt # desmonta recursivamente
reboot 
\end{lstlisting}

\vspace{\baselineskip}

Não se esqueça de executar o \texttt{mkinitcpio -P} antes de sair do ambiente arch-chroot, este comando é necessário para aplicar as instalações que precisam carregar módulos essenciais no initframs, sem esse comando o sistema pode não funcionar corretamente. Sai do ambiente arch-chroot usando \texttt{exit}, desmonte recursivamente as partições usando \texttt{umount -R /mnt} e reinicie para finalizar a segunda etapa da instalação.\\

Se você obtiver êxito nesta segunda etapa, você terá a instalação mínima do arch-linux, as etapas posteriores são considerados customização do sistema, onde você pode decidir sobre os componentes gráficos, os serviços de áudio e os programas do seu ambiente linux.\\

Se você seguiu a recomendação de não instalar o grub pelo arch-linux, então o seu sistema não estará registrado como opção. Para que a opção do arch-linux apareça é necessário inicializar o seu sistema linux principal e executar o comando para atualizar o \texttt{/boot/grub/grub.cfg}. Em sistemas \texttt{Debian} basta executar o comando \texttt{update-grub}, este comando irá executar o \texttt{os-probe} e regenerar o arquivo de inicialização incluindo todos os sistemas detectados. Caso o seu sistema seja similar ao arch-linux, você terá que fazer as operações de detecção de sistemas e regeneração apropriadas.\\

Ao reiniciar o seu recém-instalado sistema, você irá entrar no terminal virtual do kernel linux (tty), os terminais virtuais podem ser acessados usando os atalhos de teclado \texttt{ctrl+alt+F1}, \texttt{ctrl+alt+F2} ... \texttt{ctrl+alt+FX}. Cada um dos atalhos é um terminal diferente, você pode utilizar estes atalhos para fazer operações distintas no sistema. É importante saber destes atalhos, pois em caso de falha de um ambiente desktop instalado você pode voltar para a interface inicial tty, permitindo você solucionar os possíveis problemas. Como recomendação, enquanto você estiver fazendo modificações no seu sistema, sempre inicie o sistema operacional pelo terminal tty, pois ele vai permitir você resolver eventuais problemas do seu ambiente gráfico que irão surgir nesta etapa inicial, além disso você terá liberdade de modificar arquivos que podem estar bloqueados no ambiente gráfico (por exemplo, a atualização do cache de fonte). \\

\section{Configurações Básicas}

\subsection{Configurações de Rede}

\hspace{\baselineskip}

Se você instalou usando hardware wifi incompatível usando \textbf{blacklists} de drivers, você terá que criar um arquivo na pasta \texttt{/etc/modprobe.d/} do seu sistema com os drivers que você bloqueadou durante a instalação. O instalador não configura isso automaticamente: \\

\begin{lstlisting}
# 1. Entre com seu usuário wheel 
# 2. Abra o nano como administrador
# ... o arquivo deve terminar em .conf 
# ... pode ter qualquer nome 
sudo nano /etc/modprobe.d/my-blacklist.conf 

    # /etc/modprobe.d/my-blacklist.conf 
    blacklist NOME_DO_DRIVER_1
    blacklist NOME_DO_DRIVER_2

# 3. Salve o arquivo e saia do nano 
# 4. Regenere o initframs 
mkinitcpio -P 
# 5. Reinicie o seu sistema
sudo reboot
\end{lstlisting}

\vspace{\baselineskip}

O arch-linux segue a filosofia \textbf{kiss} (Keep it Simple, Stupid), seus programas não vão executar tarefas de maneira automática, o sistema não irá assumir que certas configurações sejam adequadas por conta de sua popularidade, você terá que instruir explicitamente e configurar explicitamente o que você quer. Essa filosofia é interessante para manter o sistema mínimo e personalizado. Portanto ao instalar o pacote do \texttt{NetworkManager}, o serviço provavelmente estará inativo, execute o comando \texttt{sudo systemctl enable NetworkManager} para ativar o serviço, este comando registra o serviço do \textbf{NetworkManager} para sempre ser executado ao reiniciar o sistema, portanto somente é necessário executá-lo uma vez. \\

\begin{lstlisting}
# Inventário : wifi { NetworkManager, Linux }
# 1. nmcli device status ; mostra o status das interfaces de rede 
# 2. nmcli radio wifi on ; ativa a detecção wifi 
# 3. nmcli device wifi list ; lista as redes wifis encontradas 
# 4. nmcli device wifi connect SSID password PASSWORD ; conecta a rede wifi com senha 
# 5. nmcli connection show ; mostra o nome das conexões 
# 6. nmcli connection delete NOME_DA_CONEXAO ; deleta o registro de uma conexão
# 7. nmcli connection add type ethernet con-name shared-lan ifname NOME_DA_INTERFACE ipv4.method shared ; compartilha conexão via cabo de rede com 'shared-lan' o nome da conexão 
# 8. nmcli connection up shared-lan ; ativa a conexão compartilhada acima.
\end{lstlisting}

\vspace{\baselineskip}

Com o serviço de rede funcional e o sistema conectado a internet, é interessante o usuário fazer o upgrade do sistema antes de começar qualquer instalação. O repositório oficial e o sistema do arch-linux fazem uso de update contínuo (rolling release), diferente de outros sistemas que esperam por grandes atualizações, as atualizações são entregues de imediato, conforme os desenvolvedores disponibilizem nos repositórios. Essa abordagem tem vantagens e desvantagens, a vantagem é que você sempre tem a versão mais atualizada dos recursos, a desvantagem é que isso torna o sistema instável, uma atualização quebrada pode afetar os programas que dependem dele, outro risco é de que o usuário instale um programa com dependências compartilhadas com versõe mais recentes, isso irá atualizar tais dependências, podendo afetar os programas que dependiam das versões anteriores. Execute o comando abaixo para fazer o upgrade do sistema:

$$ \texttt{sudo pacman -Syu} $$

\begin{lstlisting}
# Inventário : pacman { Arch-Linux }
# 1. sudo pacman -Syu ; upgrade de todos os pacotes
# 2. sudo pacman -Syu PACOTE1 PACOTE2 ... ; upgrade e instalação de pacote
# 3. sudo pacman -S PACOTE ; instalação de um pacote 
# 4. sudo pacman -R PACOTE ; remove um pacote 
# 5. sudo pacman -Qet ; lista os pacotes marcados como instalação manual, tome cuidado, nem todo pacote marcado como instalado manualmente foi instalado pelo usuário.
# 6. sudo pacman -Ss PACOTE ; procura um pacote no repositório oficial do arch 
# 7. sudo pacman -Qm ; lista os pacotes não oficiais 
# 8. sudo pacman -Si PACOTE ; mostra informações sobre o pacote 
# 9. sudo pacman -Qi PACOTE ; mostra informações de pacote instalado
# 10. sudo pacman -Qdt ; lista pacotes órfãos 
\end{lstlisting}

\vspace{\baselineskip}

Os sistemas de conexão wifi são regulados de maneira diferente dependendo de sua localização. Elas definem as frequências e potência permitidas por dispositivos wifi. Para entrar em conformidade com a legislação, instale o pacote \texttt{sudo pacman -S wireless-regdb}, modifique o arquivo \texttt{/etc/conf.d/wireless-regdom} e descomentando a linha com \texttt{WIRELESS\_REGDOM="BR"} para o caso brasileiro, ou o específico para sua localização. Opcionalmente você pode instalar o pacote iw com o comando \texttt{sudo pacman -Syu iw}. Ao executar o comando \texttt{iw reg get} você pode saber se a configuração do regdom está correta, e você pode configurar o regdom usando \texttt{iw reg set BR}. \\

\subsection{Configurações de Aliases}

\hspace{\baselineskip}

Os comandos linux foram padronizados em um contexto de pouca memória de trabalho e armazenamento, por isso a maior parte dos comandos são sintéticos e acrônimos. Comandos deste tipo são mais difíceis de memorizar, por isso é interessante para o usuário linux aprender a configurar o terminal para aceitar comandos customizados, os chamados \textbf{aliases}, nestes comandos recomendo não seguir o padrão sintético, mas um padrão verboso, fazendo uso da funcionalidade de auto-completar do terminal (acionado ao pressionar \texttt{tab} uma ou duas vezes).\\

Se você seguiu as instruções e criou seu usuário \texttt{wheel} com o argumento \texttt{-m}, então ao executar o comando para listar os arquivos ocultos \texttt{ls -la} do diretório \texttt{home} você vai encontrar três arquivos de configuração: \\

\begin{lstlisting}
1. .bash_profile 
2. .bashrc
3. .bash_logout
\end{lstlisting}

\vspace{\baselineskip}

O arquivo \texttt{.bash\_profile} é o arquivo principal, as instruções deste arquivo são utilizados pelos terminais e emuladores de terminais, geralmente este arquivo executa o conteúdo dos outros dois. Você pode editar diretamente o \texttt{.bashrc}, ou adicioná-los em um outro arquivo, assim como se faz nos sistemas \textbf{Debian}. Caso opte por criar um arquivo de aliases, digamos \texttt{.bash\_aliases}, você precisa editar o \texttt{.bashrc} adicionando no final do arquivo:\\

\begin{lstlisting}
if [ -f ~/.bash_aliases ]; then 
	. ~/.bash_aliases
fi
\end{lstlisting}

\vspace{\baselineskip}

Algumas sugestões de aliases que irão ajudar nesta etapa inicial: \\

\begin{lstlisting}
# write it on .bash_aliases 
alias errors='sudo dmesg -T --level=emerg,alert,crit,err,warn | tee ~/errors.txt'
alias devices='lspci -k | tee ~/devices.txt'
alias storage_devices='(sudo blkid && sudo fdisk -l) | tee ~/storage_devices.txt'
alias mount_points='sudo blkid && sudo lsblk -l'
alias update_time='timedatectl status && timedatectl set-ntp true'
alias regenerate_initramfs='sudo mkinitcpio -P'
alias regenerate_grub_configuration_file='sudo grub-mkconfig -o /boot/grub/grub.cfg'
alias regenerate_font_cache='fc-cache -f -v'
alias services='(systemctl list-units --type=service ; systemctl list-units --type=service --user)| tee ~/services.txt'
alias check_service='sudo systemctl status'
alias restart_service='sudo systemctl restart'
alias enable_service='sudo systemctl enable'
alias pacman_manually_installed='pacman -Qet | tee ~/manually_installed.txt'
alias pacman_aur_installed='pacman -Qm | tee ~/aur_installed.txt'
alias pacman_search='pacman -Ss'
alias pacman_install='sudo pacman -Syu --needed'
alias pacman_update='sudo pacman -Syu'
alias pacman_info='pacman -Si'
alias pacman_orphan='pacman -Qdt'
alias pacman_remove='sudo pacman -R'
alias pacman_uninstall='sudo pacman -R'
alias internet_device_connect='nmcli device connect'
alias internet_device_disconnect='nmcli device disconnect'

alias update_aliases='. ~/.bash_aliases'
alias edit_aliases='nano ~/.bash_aliases'
\end{lstlisting}

\vspace{\baselineskip}

Para usar de imediato os aliases você pode executar o comando \texttt{source $\sim$/.bash\_aliases}, com este comando você terá acesso aos aliases criados, como por exemplo: \texttt{show\_aliases} para mostrar todos os aliases que você criou, ou \texttt{show\_aliases | grep 'pacman'} com os aliases relacionados ao gerenciador de pacotes \textbf{pacman}.\\

Outra configuração interessante é a marcação de sintaxe do editor \textbf{nano}, em \textbf{arch-linux} a maior parte dos pacotes obedecem a filosofia \textbf{kiss}, como no caso do \texttt{nano}, que requer a criação do arquivo de configuração \texttt{.nanorc} para conseguir utilizar a marcação de sintaxe. \\

\begin{lstlisting}
# escreva em ~/.nanorc
include /usr/share/nano/*.nanorc 
# ... isso irá importar as configurações do pacote nano 
\end{lstlisting}

\vspace{\baselineskip}

Caso o arquivo \texttt{.bashrc} esteja configurado corretamente, todo novo terminal irá carregar os aliases do arquivo \texttt{.bash\_aliases}. Um dos comandos que recomendo fazer com alguma frequência nessa etapa inicial é o alias \texttt{errors} que irá executar o \texttt{dmesg} para coletar erros do sistema e armazená-los no arquivo \texttt{$\sim$/errors.txt}. Execute e abra o arquivo para verificar as mensagens de erro, nem todos os erros são importantes, se o sistema não estiver instável você pode ignorá-los, porém sempre que possível pesquise na internet por soluções destes erros.

\section{i3+picom - Ambiente Gráfico}

\subsection{Servidor Gráfico X11}

\hspace{\baselineskip}

Crie um script usando o comando \texttt{nano $\sim$/install\_xorg.sh} com o conteúdo abaixo. Esse script irá instalar os componentes essenciais do servidor gráfico X11. Salve (\texttt{ctrl+o}) e execute o comando \texttt{bash $\sim$/install\_xorg.sh}. Atenção ao caractere de quebra de linha \textbackslash, não pode haver espaço após esse caractere. \\

\begin{lstlisting}
# -- Servidor Gráfico X11
sudo pacman -Syu --noconfirm --needed \
    xorg xorg-xinit xorg-xrandr \
    xorg-xrdb xorg-twm \
    xorg-apps xterm xorg-xclock \
    ttf-dejavu ttf-liberation xorg-fonts-misc 
\end{lstlisting}

\vspace{\baselineskip}

Para inicializar o ambiente gráfico usando o \textbf{xorg}, vamos usar a ferramenta \texttt{startx} do \texttt{xinit}, porém se você tentar executar \texttt{startx} para inicializar, provavelmente o servidor irá retornar uma mensagem de inicialização e voltar para o \texttt{tty}. O motivo é a ausência de um arquivo de configuração para o \texttt{xinit}.

$$ \texttt{cp /etc/X11/xinit/xinitrc $\sim$/.xinitrc} $$

O comando acima copia a versão \texttt{default} de configuração do \texttt{.xinitrc}. Execute-o e depois execute novamente o comando \texttt{startx}. Ao executar o comando você irá entrar no gerenciador básico \texttt{twm}. Esse gerenciador será substituído pelo gerenciador \texttt{i3}, que apesar de também minimalista está sendo ativamente mantido (verifique em github.com/i3/i3).\\

\begin{lstlisting}
# -- você pode testar instalar alguns programas gráficos
sudo pacman -S --needed firefox # navegador web
sudo pacman -S --needed geany # editor de texto e código
\end{lstlisting}

\subsection{Instalação do Ambiente i3+picom}

\hspace{\baselineskip}

Você pode sair da sessão gráfica usando os atalhos \texttt{ctrl+alt+backspace} ou digitar \texttt{exit} em todos os emuladores de terminal da sessão. De volta ao \texttt{tty} vamos começar a instalação do gerenciador de janelas e o compositor. Crie um outro script de instalação com o comando \texttt{touch $\sim$/install\_i3\_picom.sh} e edite com o \texttt{nano $\sim$/install\_i3\_picom.sh}.\\

\begin{lstlisting}
# install_i3_picom.sh
sudo pacman -Syu --noconfirm --needed \
    i3-wm i3status i3lock dmenu \
    dex xss-lock network-manager-applet \
    picom
# ... não recomendo usar login gráfico em instalação customizada
# ... especialmente na distribuição arch-linux 
# sudo pacman -S --noconfirm --needed lightdm lightdm-gtk-greeter
\end{lstlisting}

\vspace{\baselineskip}

Para começar a usar o ambiente \texttt{i3} basta modificar o arquivo \texttt{$\sim$/.xinitrc} recém criado. Você deve substituir as linhas do final:

\begin{lstlisting}
...
"$twm" &
"$xclock" -geometry 50x50-1+1 &
"$xterm" -geometry 80x50+494+51 &
"$xterm" -geometry 80x20+494-0 &
exec "$xterm" -geometry 80x66+0+0 -name login
\end{lstlisting}

por:

\begin{lstlisting}
...
exec i3
\end{lstlisting}

Com isso sempre que você executar o comando \texttt{startx} no \texttt{tty}, a sessão gráfica irá chamar o gerenciador de janelas \texttt{i3}. Use o inventário abaixo para se acostumar com o ambiente: \\

\begin{lstlisting}
# Inventário : i3 { Linux }
# -- novato --
# 1. MOD+Enter // Abre um novo emulador de terminal
# 2. MOD+D // Abre o dmenu para busca de apps
# 3. MOD+<SETAS> // Muda a janela em foco
# 4. MOD+S // Muda o layout para stack

\subsection{Monitores Estendidos}

\begin{lstlisting}
# -- script para configuração de monitores estendidos 
# ... use este script depois de inicializado a sessão gráfica
# ... use o comando 'xrandr' para saber os nomes e modifique-os abaixo
DP1='NOME1'
DP2='NOME2'
# ... escolha uma das configurações para testar
xrandr --output ${DP1} --primary --auto --output ${DP2} --auto --right-of ${DP1}
xrandr --output ${DP1} --primary --auto --output ${DP2} --auto --left-of ${DP1}
\end{lstlisting}

\vspace{\baselineskip}

A configuração de monitores estendidos pode dar um pouco de dor de cabeça em placas gráficas \textbf{nvidia} ou quando temos mais de uma gpu (por exemplo, gpu integrado e a gpu dedicada de um notebook). Para ativar a funcionalidade de monitor estendido para placas gráficas nvidia é necessário que a variável do kernel \texttt{nvidia\_drm.modeset=1}. Verifique com o comando \texttt{cat /sys/module/nvidia\_drm/parameters/modeset}. Caso retorne 1 ou \texttt{Y} então ele está ativo.\\

Caso não retorne nada ou retorne \texttt{N} será necessário você mudar o parâmetro de inicialização do kernel. Vamos assumir que você esteja utilizando o grub para inicialização e que o arch-linux não seja o sistema principal (como recomendado anteriormente). A abordagem é regenerar a configuração do arquivo \texttt{/boot/grub/grub.cfg}. Lembre-se, não se edita diretamente o arquivo \texttt{/boot/grub/grub.cfg}, qualquer erro de sintaxe pode causar falha na inicialização do sistema, a abordagem é sempre editar os arquivos auxiliares: \\

\begin{lstlisting}
1. /boot/grub/grub.cfg ; arquivo auto-gerado, não edite diretamente
2. /etc/default/grub ; arquivo utilizado para configurar o grub 
3. /etc/grub.d/ ; scripts usados para configurar o grub 
\end{lstlisting}

\vspace{\baselineskip}

O arquivo (2) é o arquivo que você pode fazer edição de maneira segura. Abra o nano com privilégio de admnistrador \texttt{sudo nano /etc/default/grub}:

\begin{lstlisting}
# GRUB boot loader configuration

GRUB_DEFAULT=0
GRUB_TIMEOUT=5
GRUB_DISTRIBUTOR="Arch"
GRUB_CMDLINE_LINUX_DEFAULT="quiet"
GRUB_CMDLINE_LINUX=""

...
\end{lstlisting}

\vspace{\baselineskip}

Somente altere este arquivo no caso do \texttt{nvidia\_drm.modeset} estar desativado, nas versões recentes do arch-linux essa variável geralmente vem ativada, porém em caso de mudanças (que podem ocorrer com alguma frequência no ecossistema arch), você deve alterar a variável \texttt{GRUB\_CMDLINE\_LINUX\_DEFAULT} adicionando mais um argumento dentro das aspas \texttt{nvidia-drm.modeset=1}.

\begin{lstlisting}
...
GRUB_CMDLINE_LINUX_DEFAULT="quiet nvidia-drm.modeset=1"
...
\end{lstlisting}

\vspace{\baselineskip}

Basta agora regenerar o arquivo de configuração do grub neste sistema e no sistema principal, use o comando  \texttt{regenerate\_grub\_configuration\_file} definido no alias recomendado ou use diretamente \texttt{grub-mkconfig -o /boot/grub/grub.cfg} neste sistema. Após isso, entre no seu sistema principal e faça o update do grub. Supondo que o os-prober esteja ativo, o comando de update do sistema principal irá encontrar o arch-linux e inserir corretamente o menu do sistema arch no grub.\\

Ainda no sistema principal, você pode optar por desativar a detecção automática (que se torna lenta, quando se adiciona parâmetros de kernel customizados) e usar os blocos \texttt{menuentry} e \texttt{submenuentry} gerados do update do grub. \\

Caso queira desativar o \texttt{os-prober} (Não recomendo neste momento inicial) você deve configurar a variável do arquivo \texttt{/etc/default/grub} do sistema principal \texttt{GRUB\_DISABLE\_OS\_PROBER}, basta colocar como \texttt{true} ou colocar \# para torná-lo um comentário. Após isso você deve editar o arquivo \texttt{/etc/grub.d/40\_custom} e adicionar os menus e submenus conforme sua escolha:\\

\begin{lstlisting}
#!/bin/sh
exec tail -n +3 $0
# This file provides an easy way to add custom menu entries.  Simply type the
# menu entries you want to add after this comment.  Be careful not to change
# the 'exec tail' line above.
# ... coloque os blocos menuentry e submenuentry aqui!
### END /etc/grub.d/40_custom ###
\end{lstlisting}

\vspace{\baselineskip}

Você terá que pegar os blocos \texttt{menuentry} do arquivo \texttt{/boot/grub/grub.cfg} gerados anteriormente com o os-prober ativo (evite tentar customizar estes blocos diretamente). Lembre-se de não alterar a parte inicial e a parte final do arquivo \texttt{40\_custom} como no exemplo abaixo:\\

\begin{lstlisting}
#!/bin/sh
exec tail -n +3 $0
# This file provides an easy way to add custom menu entries.  Simply type the
# menu entries you want to add after this comment.  Be careful not to change
# the 'exec tail' line above.
menuentry 'Arch Linux (on /dev/sda5)' --class arch --class gnu-linux --class gnu --class os $menuentry_id_option 'osprober-gnulinux-simple-d268ee52-21a5-4ba8-887c-c643e62b5dde' {
	insmod part_msdos
	insmod ext2
	set root='hd0,msdos5'
	if [ x$feature_platform_search_hint = xy ]; then
	  search --no-floppy --fs-uuid --set=root --hint-bios=hd0,msdos5 --hint-efi=hd0,msdos5 --hint-baremetal=ahci0,msdos5  d268ee52-21a5-4ba8-887c-c643e62b5dde
	else
	  search --no-floppy --fs-uuid --set=root d268ee52-21a5-4ba8-887c-c643e62b5dde
	fi
	linux /boot/vmlinuz-linux root=UUID=d268ee52-21a5-4ba8-887c-c643e62b5dde rw loglevel=3 quiet lsm=landlock,lockdown,yama,integrity,apparmor,bpf
	initrd /boot/intel-ucode.img /boot/amd-ucode.img /boot/initramfs-linux.img
}
### END /etc/grub.d/40_custom ###
\end{lstlisting}

\vspace{\baselineskip}

Feito isso, basta agora regenerar o grub para aplicar o menu grub customizado. Se atente a mensagens de erro, lembre-se que sem o \texttt{grub.cfg} o grub não irá inicializar os sistemas operacionais.\\

Voltando a tarefa de configurar os monitores com placa gráfica nvidia, você deve entrar na sessão gráfica e executar os comandos abaixo:\\

\begin{lstlisting}
# -- configuração de monitores estendidos nvidia 
# lista as saídas de vídeo 
# ... procure pelas saídas conectadas 
xrandr --query
# lista os números das saídas disponíveis 
# ... anote os números dos respectivos monitores
xrandr --listproviders 
# configura o renderizador
# ... coloque o provider com o número correspondente
# ... a gpu dedicada
xrandr --setprovideroutputsource numero_fonte numero_saida
# ... basta agora executar o comando abaixo 
# ... coloque os respectivos nomes N1 N2 do xrandr --query 
xrandr --output N1 --primary --auto --output N2 --auto --right-of N1

\end{lstlisting}

\subsection{Customização - i3status}

\hspace{\baselineskip}

Os guias oficiais do i3 se encontram disponíveis em \texttt{https://i3wm.org/docs/userguide.html} para as configurações do i3 e \texttt{https://i3wm.org/docs/i3status.html} para as configurações do i3status. \\

\begin{lstlisting}
Inventário : arquivos de configuração {i3, picom}
1. ~/.config/i3/config ; crie com o comando i3-config-wizard
2. ~/.config/i3status/config ; copie de /etc/i3status.conf 
3. ~/.config/picom/picom.conf ; copie de /etc/xdg/picom.conf
\end{lstlisting}

\vspace{\baselineskip}

O arquivo (1) é o arquivo de configuração principal do i3, (2) é o arquivo de configuração da barra de status e o (3) são as configurações do compositor picom. Como personalização é pessoal por definição, o que será mostrado aqui deve ser visto como uma sugestão e um template, porém não espere customizações muito elaboradas no i3, i3+picom é um setup minimalista. \\

A primeira parte que considero mais interessante de ser customizada é a barra de status, pois ela pode mostrar informações úteis do seu sistema sem você precisar acessar programas de interface, por exemplo, status de rede, status de vpn, temperatura do processador, uso de ram, uso de espaço em disco, etc.\\

Crie o arquivo \texttt{$\sim$/.config/i3status/config} ou em qualquer outro lugar e depois execute:

$$\texttt{cat /etc/i3status.conf >> $\sim$/.config/i3status/config}$$

Isso irá copiar as configurações padrão para o seu arquivo de configuração de usuário. \\

\begin{lstlisting}
# Versão Mínima ~/.config/i3status/config

# -- configurações gerais
general {
    colors = true # mostrar cores
    interval = 5 # intervalo de atualização em segundos
}

# -- configura a ordem de display das seções
order += "disk /"
order += "tztime local" # nome do bloco entre aspas

# -- configurações das seções 
# ... os blocos não precisam estar em ordem 
# ... bloco tztime local para display de data e hora
tztime local { 
    format = "%Y-%m-%d %H:%M:%S"
}
# ... bloco uso do hd/ssd
disk "/" {
    format = "%used (%percentage_used)"
}
\end{lstlisting} 

Os blocos mais fáceis de inserir são os que não requerem buscar caminhos de diretório específicos. As variáveis \texttt{format} configuram o formato do texto que irá aparecer na barra de status, as variáveis \texttt{threshould} indicam o pontos de mudança de cor, útil para indicar uso excessivo de algum recurso de hardware, como uso intensivo de cpu ou da gpu. \\

\begin{lstlisting}
# bloco de conexão sem fio 
order += "wireless _first_"
wireless _first_ {
    format_up = "(%quality )"
    format_down = "( - )"
}

# bloco de conexão ethernet
order += "ethernet _first_"
ethernet _first_ {
    format_up = "[ %speed ]"
    format_down = "[ - ]"
}

# bloco de uso de cpu 
order += "cpu_usage"
cpu_usage {
    format = "%usage "
    format_above_degraded_threshold = "cpu: %usage "
    max_threshold = "80"
    degraded_threshold = "25"
}

# bloco de uso de memória 
order += "memory"
memory {
    format = "%used (%percentage_used)"
    threshold_degraded = "10%"
    format_degraded = "MEMORY < %available"
}

# bloco de uso de armazenamento
order += "disk /"
disk "/" {
    format = "%used (%percentage_used)"
}
\end{lstlisting}

\vspace{\baselineskip}

Porém para os outros blocos, como por exemplo, a temperatura da temperatura, você terá que procurar e testar os endereços de arquivos virtuais que mostram as leituras dos sensores. Instale o pacote \texttt{lm\_sensors} e ative seu serviço \texttt{sudo systemctl enable lm\_sensors.service}, pois eles são requeridos para ativação dos sensores.\\

\begin{lstlisting}
function searchGeneralSensorFiles {
    echo '-- /sys/class/hwmon'
    if [ -d '/sys/class/hwmon' ]; then
        for hwmon_link in /sys/class/hwmon/hwmon*; do
            if [ -d "$hwmon_link" ]; then
                sensor_name=$(cat "$hwmon_link/name" 2>/dev/null || echo "unknown")
                stable_path=$(readlink -f "$hwmon_link")
                echo "  [Dispositivo: $sensor_name]"
                echo "    -> Link: $hwmon_link"
                echo "    -> Caminho Estável: $stable_path"
                for file in "$hwmon_link"/*; do
                    if [ -f "$file" ]; then
                        echo "       $file"
                    fi
                done
            fi
        done
    fi
}    
\end{lstlisting}

\vspace{\baselineskip}

O script acima pode ajudar a encontrar os arquivos de temperatura, que geralmente tem o formato \texttt{tempX\_input}. Teste a saída destes arquivos usando \texttt{cat}, o arquivo correto de leitura de temperatura deve retornar algo como \texttt{33000}. Anote o caminho até este arquivo e use na configuração como no exemplo abaixo:\\

\begin{lstlisting}
# bloco de temperatura de cpu 
order += "cpu_temperature 0"
cpu_temperature 0 {
    format = "%degrees C"
    # format_above_threshold = ""
    path = "/sys/class/hwmon/hwmon1/temp1_input"
    max_threshold = 45
}
\end{lstlisting}

\vspace{\baselineskip}

Para o monitoramento da ativação de vpn e de montagem de partições você pode utilizar o bloco \texttt{path\_exists}. No arch-linux o endereço de montagem de partições dos discos rígidos ficam geralmente em \texttt{/run/media/user/NOME}, e para o caso do vpn você pode usar o endereço \texttt{/proc/sys/net/ipv4/conf/NOME}, substituindo \texttt{NOME} pelos diretórios e arquivos corretos. \\

\begin{lstlisting}
# bloco exemplo de vpn do protonvpn
order += "path_exists VPN"
path_exists VPN {
    format = "{vpn}"
    format_down=""
    path = "/proc/sys/net/ipv4/conf/proton0"
}

# bloco exemplo para montagem de partição 
order += "path_exists Data1"
path_exists Data1 {
    format = "[%title]"
    format_down=""
    path = "/run/media/user/Data1"
}
\end{lstlisting}

\vspace{\baselineskip}

Para que o i3 use seu arquivo de configuração você deve colocar o arquivo no endereço correto \texttt{$\sim$/.config/i3status/config} ou alternativamente modificar o arquivo de configuração \texttt{$\sim$/.config/i3/config} na seção do \texttt{i3bar}. \\

\begin{lstlisting}
...
# Start i3bar to display a workspace bar (plus the system information i3status
# finds out, if available)
bar {
    status_command i3status -c CAMINHO_DO_ARQUIVO_DE_CONF
    ...
}
...
\end{lstlisting}

\subsection{Customização - i3 geral}

\hspace{\baselineskip}

Assim como no caso da configuração do i3status, você pode colocar um arquivo de configuração em local diferente do esperado, basta editar o arquivo \texttt{.xinitrc} e colocar \texttt{exec i3 -c CAMINHO\_DO\_ARQUIVO\_DE\_CONF}.\\

No arquivo de configuração geral \texttt{$\sim$/.config/i3/config} é interessante colocar a inicialização do modo estendido de vídeo, caso o seu computador seja um desktop com disposição fixa de monitores. Basta colocar no final do arquivo de configuração a linha:\\

\begin{lstlisting}
...
# D1 e D2 devem ser substituídos pelo nome correto 
exec --no-startup-id xrandr --output D1 --primary --auto --output D2 --auto --right-of D1 
\end{lstlisting}

\vspace{\baselineskip}

Sem entrar nos detalhes, vamos nos ater a alguns pontos importantes da customização do i3. Bem no início do arquivo de configuração você pode alterar a tecla modificadora principal (que por padrão vem como a tecla da logo do windows). Para saber os keycodes e os nomes das outras teclas, basta usar o comando interativo \texttt{xev}.\\

Na parte da tela de bloqueio \texttt{lock-screen}, o padrão é uma tela cinza, você pode criar um script customizado para gerar uma imagem desfocada da tela.\\

\begin{lstlisting}
#!/bin/bash
# Screenshot 
scrot /tmp/lockscreen.png
# Desfoque de imagem 
magick /tmp/lockscreen.png -blur 0x5 /tmp/lockscreen_blur.png
# Usa o i3lock para bloquear a tela com a imagem gerada
i3lock -i /tmp/lockscreen_blur.png
# Apaga as imagens
rm /tmp/lockscreen.png /tmp/lockscreen_blur.png
\end{lstlisting}

\vspace{\baselineskip}

Será necessário instalar os pacotes adicionais \texttt{scrot} e \texttt{magick} para usar o script acima. Criado o arquivo, basta substituir a linha do arquivo de configuração:\\

\begin{lstlisting}
# ... substitua 
exec --no-startup-id xss-lock --transfer-sleep-lock i3lock --nofork
# ... por 
exec --no-startup-id xss-lock --transfer-sleep-lock ~/.config/i3/NOME_DO_ARQUIVO.sh
\end{lstlisting}

\vspace{\baselineskip}

A flag \texttt{--no-startup-id} serve para o i3 não gerenciar o comando, os comandos gerenciados podem ser usados por exemplo para definir workspaces específicos para determinado tipo de aplicação. Porém alguns aplicativos não são compatíveis, então a flag serve para explicitar que o comando não será gerenciado pelo i3. Para definir execuções de comando o usuário pode utilizar \texttt{exec} que executa o comando apenas uma vez na inicialização, ou \texttt{exec-always}, que executa o comando sempre que iniciado ou reiniciado. O comando \texttt{bindsym} serve para definir um comando via atalho de teclado, o usuário pode definir um atalho para uma funcionalidade customizada, ou mesmo um script como abaixo: \\

\begin{lstlisting}
# Exemplo de comando de teclado:
# ... kill-switch para conexão da internet
bindsym F12 exec nmcli networking off
\end{lstlisting}

\begin{lstlisting}

\vspace{\baselineskip}

Alguns aplicativos tem sua inicialização gerenciada pelo xdg através de arquivos \texttt{.desktop}, como por exemplo o compositor picom (verifique a lista dos aplicativos gerenciados com o comando \texttt{dex -ad}). Outros aplicativos podem ser chamados via o arquivo de configuração principal do i3 da mesma forma que comandos, como o comando de monitor estendido feito anteriormente. \\

Você também pode chamar scripts customizados, por exemplo, um script para mudar a imagem de fundo aleatóriamente em intervalos de tempo definidos. Crie o arquivo com conteúdo abaixo: \\

\begin{lstlisting}
#!/bin/bash
# random_background.sh
# Seleciona backgrounds aleatórios continuamente.
# Coloque em ~/Pictures/Backgrounds
BACKGROUND_DIRECTORY="$HOME/Pictures/Backgrounds"
while true; do
    # I := Se o diretório existir e não estiver vazio
    if [ -d "$BACKGROUND_DIRECTORY" ] && \
       [ -n "$(find "$BACKGROUND_DIRECTORY" -type f 2>/dev/null)" ]; then
        # R := Seleciona aleatoriamente uma imagem como background
        RANDOM_BACKGROUND="$(find "$BACKGROUND_DIRECTORY" -type f | shuf -n 1)"
        feh --bg-fill "$RANDOM_BACKGROUND"
    fi
    # S := Pause por 10 minutos
    sleep 600
done
\end{lstlisting}

\vspace{\baselineskip}

Depois insira uma linha exec apontando para este arquivo, dentro do arquivo principal de configuração i3:\\

\begin{lstlisting}
# tome cuidado para o script/programa não bloqueie a execução do i3
# ... lembre-se de colocar & no final para chamá-lo como subprocesso
# ... independente.
exec --no-startup-id ~/.config/i3/random_background.sh &
\end{lstlisting}

\subsection{Customização - picom}

\hspace{\baselineskip}

O compositor picom permite colocar efeitos de sombra, transição, transparência e desfoque. Para configurar o picom você pode utilizar o arquivo \texttt{/etc/xdg/picom.conf} como base. Crie uma cópia usando os comandos abaixo: 

$$ \texttt{mkdir -p $\sim$/.config/picom} $$
$$ \texttt{cp /etc/xdg/picom.conf $\sim$/.config/picom/} $$

\vspace{\baselineskip}

Abaixo temos um exemplo de configuração mínima.\\

\begin{lstlisting}
# -- Backend --
# 1. glx é mais estável, egl experimental
# 2. xrender é o backend legado 
backend = "glx" ; # glx, egl, xrender, xr_glx_hybrid 
vsync = true; # sincronia vertical
use-damage = true; # otimização

# -- Opacidade/Transparência --
# 1. use `xprop WM_CLASS` para obter os nomes das janelas
# 2. use opacity-rule para excluir janelas específicas
# ... por exemplo players de vídeo e browsers

\vspace{\baselineskip}

No caso do ambiente arch-linux, além dos repositórios oficiais, existem os repositórios mantidos por usuários (AUR, arch-linux user repository) com ferramentas de gerenciamento similar a \texttt{pacman}. Tais repositórios são tão confiáveis quanto repositórios do github (ou seja, não são confiáveis), você precisa estar atento a mundaças e confiar no autor de tais repositórios. Evite repositórios abandonados por muito tempo, evite update automático de repositórios \texttt{AUR} e se possível busque programas nos repositórios oficiais em vez dos repositórios da comunidade.\\

\begin{lstlisting}
Práticas de Segurança Linux 
1. Dê preferência por instalação via repositórios oficiais ; Repósitórios oficiais da sua distribuição linux são regularmente mantidos pela distribuição, verificando assinaturas digitais e mantendo dependências seguras.
2. Ao utilizar repositórios de terceiros verifique as assinaturas GPG.
3. Atualize regularmente o sistema para obter correções e atualizações de segurança.
4. Evite instalação via script .sh ; Usar scripts para instalação é uma má prática. Algumas fontes conhecidas usam scripts deste tipo, porém você deve ler atentamente o script e evitar fontes desconhecidas. 
5. Sempre tente executar uma tarefa primeiro como usuário normal ; Boa parte das tarefas, incluso algumas instalações, podem ser feitas como usuário normal, use sudo somente se necessário.
6. Reputação e Auditoria ; Um repositório/aplicativo ser de código-aberto não implica automaticamente em segurança, é necessário que auditorias sejam feitas regularmente e tenha uma comunidade ativa. É muito arriscado usar repositórios pouco conhecidos ou abandonados. 
7. Compilação a partir da fonte ; Se o aplicativo for de código aberto, é possível optar por binários já pré-compilados. Geralmente os repositórios disponibilizam as checksums para fazer a comparação, porém a compilação a partir da fonte costuma ser mais segura. 
8. Checksums ; O binário de um aplicativo pode ser comparado com o original usando o recurso checksum. Checksums são algoritmos que retornam um valor numérico a partir de um arquivo binário, a ideia é que para um mesmo aplicativo temos apenas um valor de checksum para um mesmo algoritmo, então é possível verificar se um aplicativo é autentico ou se não foi alterado ou corrompido. 
\end{lstlisting}

\vspace{\baselineskip}

Lembre-se, \texttt{AUR} não é repositório oficial, ele é mantido pela comunidade arch-linux, deve ser visto com cautela assim como os repositórios do github. Evite atualizações automáticas, evite usar repositórios abandonados e sempre que possível verifique por notícias de segurança da comunidade arch-linux em seu site oficial. \\

Os gerenciadores de pacote do \texttt{AUR} não vem instalados e nem estão nos repositórios oficiais, um gerenciador bem conhecido \texttt{yay} pode ser instalado via \textbf{github}.








\newpage

Um aplicativo de código-aberto não se torna automaticamente seguro, é necessário haver uma comunidade de programadores para fazer auditorias de código de maneira regular. Projetos pequenos ou abandonados podem ser um risco de segurança para o usuário. \\

















Muitos aplicativos de código-aberto disponibilizam binários pré-compilados, o que torna mais prático para projetos grandes, pois a compilação a partir da fonte exige algum conhecimento de programação além de consumir tempo e processamento do computador. Para verificar se os binários pré-compilados correspondem ao resultado da compilação as distribuições linux vem equipadas com comandos para calcular checksums. Esses comandos geram um número (hash) a partir do conteúdo do próprio arquivo binário, então um mesmo arquivo binário gera apenas um único número para um mesmo algoritmo. Seria ideal que para um mesmo algoritmo dois binários diferentes gerassem valores diferentes, mas não é sempre isso que acontece, então esse método deve ser interpretado da seguinte forma, binários adulterados tem maiores chances de gerarem valores diferentes dependendo do algoritmo. Quando dois binários geram um mesmo hash dizemos que há colisão. \\

\begin{lstlisting}
# Inventário [ Checksums ] { Linux, Debian }
# 1. sha256sum <ARQUIVO> # SHA-256 (256-bit) - Checksum Padrão, com chance de colisão estatisticamente irrelevante. 
# 2. md5sum <ARQUIVO> # MD5 (128-bit) - Não é mais utilizado para autenticação por ser fácil forjar colisão de hash com ferramentas atuais. Porém pode ser usado para verificar modificações e corrupções durante cópia e transferência de arquivos. 
# 3. sha1sum <ARQUIVO> # SHA-1 (160-bit) - Não pode mais ser utilizado para autenticação, porém pode ser usado para verificar modificações e corrupções durante cópia e transferência de arquivos.
# 4. sha384sum <ARQUIVO> # SHA-384 (384-bit) - Mais forte que SHA-256.
# 5. sha512sum <ARQUIVO> # SHA-512 (512-bit) - Hash maior, mais forte que SHA-256.
\end{lstlisting}

\vspace{\baselineskip}

Para verificar autenticidade geralmente você irá comparar o sha256sum gerado pelo seu arquivo com o do distribuidor. Para verificar corrupção não intencional de arquivos durante transferência ou cópia você pode utilizar o md5sum ou sha1sum que podem detectar erros de transferência. \\
